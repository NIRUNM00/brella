# Brella — AI 出片策展 Agent

> 你生成，我来筛。不用改工作流，不靠玄学评分。

---

## 一句话定位

**为重度 AI 图像生成者设计的策展助手**——通过 Briefing→Detail→Decision 交互协议，记住你的审美偏好，从海量出图中帮你找到真正要留的片子。

---

## 痛点

AI 生图正在制造一个喜剧性的鸿沟：

- 一个 prompt 跑 4-8 张变体，8 种种子 x 3 种 CFG = **24 张/轮**
- 一轮 24 张，跑了 20 组 variant = **480 张**
- 你要在 480 张里手工一张张翻，凭记忆回想起 "第二轮的 seed 42 那个构图我好像喜欢"

**结果**：花在筛图上的时间 ≈ 花在生图上的时间。

现有工具的解法是「AI 自动评分」——但评分模型不认识你，它不知道你喜欢手部过曝还是构图倾斜。而且你也不信任它的评分。

**真实需求不是打分，是记得你上次选了什么、为什么选。**

---

## 解决方案

### Briefing→Detail→Decision 协议

这不是另一个批处理脚本。这是一个 **交互决策引擎**：

```
你生成 200 张 → Brella 给你一份简报
               ("本批 200 张，Bad 层 32 张（崩手/崩脸），
                Dubious 层 128 张（构图偏/光线过曝），
                Desired 层 40 张（无缺陷/构图工整）")
你：感兴趣 Desired，但 Dubious 里某个 variant 看看 →
Brella 拉出该 variant 的 4 张种子详情（seed=42,77,108,203）
你：seed 77 构图可以但CFG太高，留seed42和108 →
Brella 记录偏好 ✅ 记住了 + 更新 Wilson 分数
```

**三层分类（Bad / Dubious / Desired）**

| 层级 | 含义 | 处理 |
|------|------|------|
| Bad | 崩手/崩脸/结构缺陷 | 默认淘汰，不接受反驳 |
| Dubious | 构图/光线/风格有疑问 | 按 Wilson Score 排序，推荐优先审 |
| Desired | 无明显缺陷，构图可用 | 高概率保留，按偏好排序 |

## 核心差异化

### ✅ 竞品全部空白的三块

| 能力 | Brella | Aftershoot | FilterPixel | Imagen |
|------|--------|-----------|-------------|--------|
| **跨批次审美记忆** | ✅ SQLite 持久化 | ❌ | ❌ | ⚠️ 仅编辑风格 |
| **种子-提示词关联** | ✅ | ❌ | ❌ | ❌ |
| **Wilson Score 优先推荐** | ✅ | ❌ | ❌ | ❌ |
| **B→D→D 交互协议** | ✅ 定义级 | ❌ | ❌ | ❌ |

### 更精确的差异

- **不是评分竞赛** —— 不追求评分模型准确率，追求"你真的会用它决策"
- **不是黑盒** —— 每个建议都有 Detail 可回溯，不是"AI 说这张 87 分"
- **记忆力** —— 上次选过什么、为什么选，下次出图自动带入

## 架构一览

```
┌─────────────────────────────────────────────┐
│              B→D→D 协议引擎                   │
│  • Briefing 生成层（聚合→简报）                │
│  • Detail 检索层（种子↔图像↔元数据）            │
│  • Decision 记录层（偏好写入 SQLite）          │
├─────────────────────────────────────────────┤
│              跨批次记忆 (SQLite)              │
│  • seed_preferences: seed ↔ accept/reject    │
│  • prompt_archetypes: prompt ↔ 种子偏好模式    │
│  • image_metadata: seed + prompt + cfg + model│
├─────────────────────────────────────────────┤
│              Wilson Score 引擎                │
│  • 置信度加权排序                             │
│  • 样本少的种子自动降权                        │
├──────────────────┬──────────────────────────┤
│  ComfyUI 适配层   │  摄影适配层 (预留)         │
│  • 节点集成       │  • EXIF/RAW 解析          │
│  • PNG 元数据读取  │  • Lightroom Plugin       │
│  • 工作流挂钩      │  • 风格预设映射            │
└──────────────────┴──────────────────────────┘
```

**关键设计原则**：协议核心层是全域通用的，适配层按 domain adapter 模式隔离。ComfyUI 先行，摄影扩展只写一个 adapter。

## 目标用户

**第一梯队（MVP 验证期）**

> 重度 AI 图像生成者，每周产出 100-1000+ 张
> 使用 ComfyUI 出图，习惯批量跑变体
> 对出图质量有高标准，不愿接受自动评分的结果
> 技术能力足以安装一个 ComfyUI 自定义节点

**第二梯队（扩展期）**

> 专业摄影师（婚礼/活动/人像）
> 每场拍摄 1000+ raw 照片需要初步筛选
> 使用 Lightroom / Capture One
> 愿为自动化筛选付费（参考 Aftershoot $10-60/月）

---

## MVP 范围（约 6 周）

| 阶段 | 内容 | 交付 |
|------|------|------|
| W1-2 | B→D→D 协议核心 + 终端 UI | CLI 可用，支持手动加载目录 → 分层 → 决策 |
| W3-4 | 跨批次记忆 SQLite schema + CRUD | 种子偏好持久化，跨 session 生效 |
| W5-6 | ComfyUI 自定义节点集成 | 节点读取 PNG 元数据 + 调用 Brella 引擎 |
| 并行 | 轻量质量检测管道（Bad 层判定） | 手部/面部结构检测 + 构图/曝光基础评分 |

**非 MVP 范围**：摄影适配层、云端部署、团队协作、AI 自动拒绝

## 竞品地图

```
                 手动筛选                      自动筛选
                  │                            │
                  │       Imagen AI             │
  高定制化 ────────┤─ Aftershoot ──────────────┤─ Brella (B→D→D)
                  │  FilterPixel              │  (跨批次记忆)
                  │  Narrative Select          │
                  │                            │
  低定制化 ────────┤───────────────────────────┤─ 通用评分模型
                  │                            │
                  │    无种子关联               │  有种子关联
                  │    无跨批次记忆             │  有跨批次记忆
```

**现状**：市场上的筛选工具都在自动评分赛道竞争，**没有人做交互式策展 + 审美记忆**。这是 Brella 的位置。

---

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| Aftershoot 可能快速跟进记忆功能 | 先发优势 + Wilson Score 算法壁垒 + ComfyUI 社区粘性 |
| ComfyUI 节点安装门槛高 | 提供一键安装脚本 + 详细文档 |
| 用户习惯难改变（已习惯手动筛） | 渐进式——不取代现有流程，只减少翻图次数 |
| 二手/ACG 图像评估模型精度不足 | B→D→D 协议中用户做最终决定，模型只做"建议" |
